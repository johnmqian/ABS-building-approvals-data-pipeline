# ABS Building Approvals Pipeline

An end-to-end data pipeline that ingests Australian Bureau of Statistics (ABS) Building Approvals data at the SA2 geographic level, lands it in Amazon S3, transforms it through a Medallion (Bronze / Silver / Gold) architecture, and serves the gold layer via Snowflake and dbt. Orchestration is handled by Apache Airflow running in Docker.

## Pipeline Stages

### 1. `generate_params` (PythonOperator)

Generates two stable values shared across all downstream tasks:

| XCom key | Value | Example |
|---|---|---|
| `ingestion_date` | Airflow `logical_date` converted to Melbourne time (`YYYY-MM-DD`) | `2026-03-19` |
| `run_id` | Airflow's built-in unique run identifier | `manual__2026-03-19T...` |

Using Melbourne time here (once) ensures Bronze, Silver, and Gold always agree on the date even if a run crosses midnight.

`start_period` and `end_period` are DAG params (not XCom) вҖ” they are known before the run starts and can be overridden in the Airflow trigger UI without touching code.

---

### 2. Bronze вҖ” `run_bronze_extraction_ingestion` (BashOperator)

**Script:** `ingestion/ABS_Building_Approvals_Extraction.py`

Runs as a separate OS process (not inside the Airflow worker process) to avoid loading heavy ETL dependencies into Airflow's task runner. Configuration is injected as environment variables via the BashOperator `env` parameter.

**What it does:**
1. Reads SA2 code lists from `ingestion/2021_SA2_lists/` вҖ” 16 region files covering all Australian capital cities and rest-of-state areas.
2. For each region, builds an ABS SDMX REST API URL and fetches data as CSV:
   - **Dataset:** `ABS,BA_SA2,2.0.0` (Building Approvals by SA2)
   - **Period:** Controlled by `start_period` / `end_period` DAG params (defaults: `2021-07` вҖ“ `2026-01`)
   - **Building types:** `100, 110, 120, 121, 122, 130, 131, 132, 133, 134, 150`
   - **Measures:** Number of dwellings, value of work, etc.
3. Standardises column names (snake_case) and parses types (via `ingestion/functions.py`).
4. Writes one Parquet file per region to a temporary directory, then uploads to S3.
5. Before writing, deletes any existing objects under the target prefix (idempotent overwrite).
6. Raises an error if zero total rows are written across all regions.

**HTTP resilience:** 5 retries with exponential backoff; retries on 429/500/502/503/504.

**S3 output:**
```
s3://johnq-data-lake-dev/abs-building-approvals/bronze/
в””в”Җв”Җ ingestion_date=YYYY-MM-DD/
    в”ңв”Җв”Җ part-00000.parquet   # Greater_Sydney
    в”ңв”Җв”Җ part-00001.parquet   # Greater_Melbourne
    в””в”Җв”Җ ...                  # one file per region
```

**Bronze schema:**

| Column | Type | Description |
|---|---|---|
| `time_period` | date | Observation month |
| `obs_value` | float64 | Observed value |
| `obs_status` | string | Observation status code/label |
| `obs_comment` | string | Observation comment |
| `dataflow` | string | ABS dataflow identifier (`code: label`) |
| `measure` | string | Measure descriptor (`code: label`) |
| `sector` | string | Sector of ownership (`code: label`) |
| `work_type` | string | Type of work (`code: label`) |
| `building_type` | string | Type of building (`code: label`) |
| `region_type` | string | Region type (`code: label`) |
| `region` | string | SA2 region (`code: label`) |
| `frequency` | string | Frequency (`code: label`) |
| `unit_measure` | string | Unit of measure (`code: label`) |
| `unit_mult` | string | Unit multiplier (`code: label`) |
| `region_file` | string | Source region file name |
| `ingestion_date` | string | Pipeline run date (`YYYY-MM-DD`) |
| `run_id` | string | Airflow run ID |

---

### 3. Silver вҖ” `run_silver_databricks` (DatabricksRunNowOperator)

**Notebook:** `databricks/notebooks/10_silver_abs_building_approvals.ipynb`
**Databricks Job ID:** `1099697880350402`
**Compute:** Serverless

Reads the Bronze Parquet for the given `ingestion_date` and produces a cleaned Delta table.

**Transformations:**
- Splits all `code: label` combined columns into separate `_code` and `_label` columns (e.g. `building_type` вҶ’ `building_type_code`, `building_type_label`).
- Casts `obs_value` to `double`.
- Drops the raw combined columns.

**Write pattern:** `overwrite` with `replaceWhere ingestion_date = 'YYYY-MM-DD'` вҖ” only the current partition is replaced, preserving all previous runs.

**S3 output:**
```
s3://johnq-data-lake-dev/abs-building-approvals/silver/   (Delta table, partitioned by ingestion_date)
```

---

### 4. Load Silver to Snowflake вҖ” `run_load_silver_snowflake` (SnowflakeOperator)

Runs a `COPY INTO` statement that loads the silver Parquet files from S3 into the Snowflake staging table. Snowflake tracks load history per file вҖ” files already loaded are automatically skipped, preventing duplicates.

**Snowflake objects:**
- **Stage:** `ABS_BUILDING_APPROVALS_DATABASE.ABS_BUILDING_APPROVALS.S3_SILVER_STAGE` вҖ” external stage pointing to the silver S3 path
- **Target table:** `ABS_BUILDING_APPROVALS_DATABASE.ABS_BUILDING_APPROVALS.SILVER_ABS_BUILDING_APPROVALS`

---

### 5. Gold вҖ” `run_gold_dbt` (BashOperator вҶ’ dbt)

**dbt model:** `dbt/models/gold/gold_abs_building_approvals.sql`

Runs `dbt run --select gold_abs_building_approvals` inside the container. dbt connects to Snowflake via `profiles.yml` (mounted at `/home/airflow/.dbt/profiles.yml`) and executes the gold model SQL directly in Snowflake.

**Transformations:**
- Groups by `time_period`, `region_code`, `region_label`, `building_type_label`, `measure_label`, `sector_label`, `work_type_label`, `ingestion_date`.
- Sums `obs_value` across all SA2s within each group.

**Snowflake output:**
```
ABS_BUILDING_APPROVALS_DATABASE.ABS_BUILDING_APPROVALS.GOLD_ABS_BUILDING_APPROVALS
```

## Setup & Running

### Prerequisites

- Docker + Docker Compose
- AWS credentials with S3 read/write access to `johnq-data-lake-dev`
- Databricks workspace with job `1099697880350402` (Silver) configured
- Snowflake account with the objects created via `dbt/setup/SQL_setup_commands.sql`

### 1. Configure environment

Create `airflow/.env` (do **not** commit to source control):

```env
AWS_ACCESS_KEY_ID=<your-key>
AWS_SECRET_ACCESS_KEY=<your-secret>
```

Create `airflow/secrets/profiles.yml` (do **not** commit to source control) with your Snowflake connection details. Use the same format as `~/.dbt/profiles.yml` generated by `dbt init`.

### 2. Set up Snowflake objects

Run the SQL in `dbt/setup/SQL_setup_commands.sql` in the Snowflake UI (Worksheets) to create the warehouse, database, schema, staging table, and S3 external stage.

### 3. Build and start Airflow

```bash
cd airflow
docker compose up --build -d
```

The web UI will be available at http://localhost:8080 (default credentials: `airflow` / `airflow`).

### 4. Configure Airflow connections

In the Airflow UI (**Admin вҶ’ Connections**), add:

**Databricks connection:**
- **Conn ID:** `databricks_default`
- **Conn Type:** Databricks
- **Host:** your Databricks workspace URL
- **Token:** your Databricks personal access token

**Snowflake connection:**
- **Conn ID:** `snowflake_default`
- **Conn Type:** Snowflake
- **Login:** your Snowflake username
- **Password:** your Snowflake password
- **Extra:** `{"account": "<account_id>", "warehouse": "ABS_BUILDING_APPROVALS_WAREHOUSE", "database": "ABS_BUILDING_APPROVALS_DATABASE", "role": "SYSADMIN"}`

### 5. Trigger the DAG

The DAG (`abs_building_approvals_pipeline_w_snowflake_DAG_v1`) has no schedule and must be triggered manually from the Airflow UI.

## Key Design Decisions

- **Bronze runs externally (BashOperator):** The extraction script is kept outside the Airflow worker process to avoid loading heavy dependencies (pandas, pyarrow, boto3) into Airflow's task runner, and to make it easy to run and debug independently of Airflow.
- **Pure functions separated into `functions.py`:** Transformation logic is isolated from config and I/O, making it independently testable without env vars or AWS access.
- **Single source of truth for run parameters:** `generate_params` computes `ingestion_date` and `run_id` once and distributes them via XCom, so all three layers always agree on the date.
- **Date range as DAG params:** `start_period` and `end_period` are DAG-level params with defaults, overridable in the Airflow trigger UI вҖ” no code change needed when new ABS data is released.
- **Idempotent writes:** The Bronze S3 prefix delete-and-rewrite and the Silver `replaceWhere` Delta write mean the pipeline can be safely re-run for the same date without duplicating data. Snowflake's `COPY INTO` load history provides the same guarantee for the silver staging table.
- **Melbourne timezone:** `ingestion_date` is derived from Airflow's `logical_date` converted to `Australia/Melbourne` time, reflecting when the data logically belongs rather than UTC wall-clock time.
- **Gold moved from Databricks to Snowflake + dbt:** The gold aggregation is a simple GROUP BY вҖ” better suited to Snowflake SQL than a Spark notebook. dbt provides dependency tracking, testing, and auto-generated documentation on top of plain SQL. This also eliminates the need for a Databricks gold job, reducing compute costs.
- **dbt `profiles.yml` mounted via Docker volume:** Snowflake credentials are kept outside the image and outside the repo, mounted into the container at `/home/airflow/.dbt/profiles.yml` at runtime вҖ” the same pattern as `airflow/.env` for AWS credentials.
